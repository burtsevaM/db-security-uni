## Сценарий A: Стартап с 5 сотрудниками

### Какая модель доступа подойдет на данном этапе?

Для данного сценария наиболее подходящей будет модель DAC.  
Команда маленькая, сотрудники доверяют друг другу, структура пока простая, а доступы нужно выдавать быстро и без сложной бюрократии. В таких условиях удобно назначать права напрямую конкретным пользователям на конкретные объекты базы данных.

### Преимущества этой модели

- простота настройки и управления;
- высокая гибкость;
- можно быстро выдать или изменить доступ;
- хорошо подходит для небольшой команды без сложной иерархии.

### Недостатки этой модели

- права могут назначаться несистемно;
- при росте числа сотрудников доступами становится сложнее управлять;
- выше риск выдать лишние права;
- сложнее контролировать, кто и к чему реально имеет доступ.

### Что изменится при масштабировании команды?

Когда команда вырастет, DAC начнет хуже подходить для управления доступом.  
При большом числе сотрудников и ресурсов ручная выдача прав станет неудобной и приведет к ошибкам. На этапе роста компании разумно перейти к RBAC, чтобы назначать права не каждому человеку отдельно, а по ролям: разработчик, аналитик, тестировщик, менеджер и так далее.

### Вывод

На текущем этапе для стартапа лучше использовать DAC, потому что эта модель проста и удобна для маленькой команды.  
Но при расширении компании ее желательно заменить или дополнить ролевым подходом RBAC.

## Сценарий B: Военное ведомство

### Какая модель доступа обязательна для этого сценария?

Для этого сценария обязательна модель MAC.  
В описании прямо указаны уровни допуска и строгая регламентация доступа, а это признаки мандатной модели. В такой системе доступ определяется не желанием пользователя и не владельцем данных, а централизованной политикой безопасности.

### Как реализовать правила "No Read Up" и "No Write Down"?

Для реализации этих правил каждому пользователю и каждому объекту базы данных нужно назначить метку безопасности.

Пример уровней:
- Открыто
- Секретно
- Совершенно секретно

Тогда правила работают так:

- **No Read Up** - пользователь не может читать данные более высокого уровня, чем его допуск.  
  Например, сотрудник с уровнем "Секретно" не должен читать данные уровня "Совершенно секретно".

- **No Write Down** - пользователь не должен записывать данные на более низкий уровень.  
  Это нужно для того, чтобы секретная информация не попала в менее защищенную область.

### Какие дополнительные меры безопасности необходимы?

- централизованное управление метками доступа;
- жесткое разграничение прав по уровням допуска;
- обязательная аутентификация пользователей;
- аудит всех обращений к данным;
- принцип минимально необходимых привилегий;
- регулярная проверка прав доступа;
- защита от несанкционированного копирования и выгрузки данных.

### Вывод

Для военного ведомства подходит только MAC, так как здесь критична строгая защита конфиденциальной информации.  
Именно эта модель лучше всего обеспечивает работу с уровнями секретности и не допускает произвольного изменения прав доступа.

## Сценарий C: Корпорация с 500 сотрудниками

### Какая модель доступа оптимальна для этой организации?

Для такой организации оптимальной будет модель RBAC.  
В компании много сотрудников, отделов и уровней управления, поэтому выдавать права каждому пользователю отдельно неудобно. Намного эффективнее назначать права по ролям, связанным с должностными обязанностями.

### Как отразить организационную структуру в модели доступа?

Организационную структуру можно представить через иерархию ролей.

Пример:
- сотрудник отдела продаж;
- менеджер отдела продаж;
- директор отдела продаж;
- сотрудник HR;
- менеджер HR;
- бухгалтер;
- главный бухгалтер;
- системный администратор;
- специалист поддержки.

У каждой роли должен быть свой набор прав.  
Старшие роли могут наследовать часть прав младших, если это требуется по логике работы. Например, менеджер может иметь все базовые права сотрудника своего отдела и дополнительные управленческие права.

### Как реализовать разделение обязанностей (SoD)?

Разделение обязанностей нужно для того, чтобы один человек не мог выполнить весь критически важный процесс самостоятельно.

Примеры:
- сотрудник, который создает платеж, не должен сам его утверждать;
- разработчик не должен иметь те же права, что и администратор промышленной базы данных;
- сотрудник HR не должен одновременно менять кадровые данные и утверждать финансовые выплаты;
- оператор поддержки не должен иметь доступ к административным функциям безопасности.

Это реализуется через:
- разделение ролей по функциям;
- запрет опасных сочетаний ролей;
- выдачу только тех прав, которые реально нужны для работы;
- регулярный пересмотр ролевой модели.

### Вывод

Для крупной корпорации лучше всего подходит RBAC, потому что эта модель масштабируется и хорошо соответствует организационной структуре.  
Она упрощает администрирование, снижает риск ошибок и позволяет удобно реализовать разделение обязанностей.
