# Политика безопасности базы данных

## 1. Общие положения

### 1.1. Цель документа
Настоящий документ описывает политику безопасности базы данных `task_management`, разработанной в рамках практической работы по дисциплине «Безопасность баз данных». Документ определяет используемую модель управления доступом, роли пользователей, иерархию ролей, матрицу привилегий, ограничения на уровне строк и порядок управления доступом.

### 1.2. Область применения
Политика распространяется на базу данных `task_management`, схему `app`, таблицы `app.users`, `app.projects`, `app.tasks`, `app.comments`, `app.task_history`, `app.access_logs`, а также на роли и login-пользователей, созданных для разграничения доступа в PostgreSQL.

### 1.3. Определения и термины
- **RBAC** — ролевая модель управления доступом.
- **RLS** — ограничения доступа на уровне строк.
- **Роль** — набор привилегий, назначаемый пользователю или другой роли.
- **Привилегия** — разрешение на выполнение определенной операции над объектом БД.
- **Схема** — логическая группа объектов базы данных.
- **Политика RLS** — правило, определяющее, какие строки доступны пользователю.

## 2. Модель управления доступом

### 2.1. Выбранная модель
В базе данных используется ролевая модель управления доступом **RBAC**.

### 2.2. Обоснование выбора
Модель RBAC выбрана потому, что она позволяет:
- удобно объединять права по ролям;
- настраивать наследование прав;
- централизованно управлять доступом;
- разграничивать права сотрудников, менеджеров, администраторов и других категорий пользователей.

### 2.3. Описание ролей
В системе реализованы следующие роли:
- `app_guest`
- `app_employee`
- `app_manager`
- `app_admin`
- `app_superuser`

Для тестирования также созданы login-пользователи:
- `demo_guest`
- `demo_employee`
- `demo_manager`
- `demo_admin`
- `demo_superuser`

## 3. Описание ролей

### 3.1. Роль `app_guest`
**Назначение:** минимальный доступ к системе.  
**Права:** подключение к базе данных, использование схемы `app`, просмотр таблицы `app.projects`.  
**Ограничения:** нет доступа к таблицам `users`, `tasks`, `comments`, `task_history`, `access_logs`; нет прав на изменение данных.

### 3.2. Роль `app_employee`
**Назначение:** работа сотрудника с задачами и комментариями.  
**Права:** чтение разрешенных данных о пользователях, чтение проектов, задач, комментариев и истории изменений; добавление и обновление задач; добавление комментариев.  
**Ограничения:** нет доступа к чувствительным столбцам, например к `password_hash`; нет права удалять задачи; после включения RLS сотрудник видит только свои задачи и может изменять только свою запись в `app.users`.

### 3.3. Роль `app_manager`
**Назначение:** управление проектами и задачами в зоне ответственности менеджера.  
**Права:** наследует права сотрудника; может добавлять, изменять и удалять проекты; имеет расширенный доступ к задачам, комментариям и истории изменений.  
**Ограничения:** не управляет пользователями; после включения RLS работает только с задачами своих проектов.

### 3.4. Роль `app_admin`
**Назначение:** администрирование прикладной части БД.  
**Права:** наследует права менеджера; имеет полный доступ ко всем таблицам и последовательностям схемы `app`; может изменять данные пользователей; может создавать объекты в схеме `app`.  
**Ограничения:** роль является верхней административной ролью приложения, но не обязательно системным суперпользователем PostgreSQL.

### 3.5. Роль `app_superuser`
**Назначение:** максимальный уровень доступа в рамках приложения.  
**Права:** полный доступ к базе `task_management`, схеме `app`, таблицам и последовательностям.  
**Ограничения:** отсутствуют в рамках прикладной модели доступа.

## 4. Иерархия ролей

### 4.1. Диаграмма наследования
Иерархия ролей имеет следующий вид:

`guest -> employee -> manager -> admin -> superuser`

или сверху вниз:

- `app_superuser`
- `app_admin`
- `app_manager`
- `app_employee`
- `app_guest`

### 4.2. Описание отношений
Наследование реализовано следующим образом:
- `app_employee` наследует `app_guest`
- `app_manager` наследует `app_employee`
- `app_admin` наследует `app_manager`
- `app_superuser` наследует `app_admin`

Такая схема позволяет постепенно расширять права доступа по мере повышения роли.

## 6. Политики RLS

### 6.1. Назначение RLS
Обычные привилегии `GRANT` ограничивают доступ на уровне таблиц и столбцов. Для более точного разграничения доступа в системе дополнительно применяется **Row-Level Security**, который ограничивает доступ к отдельным строкам.

### 6.2. RLS для таблицы `app.tasks`
Для таблицы `app.tasks` реализованы политики:
- сотрудник видит только свои задачи;
- сотрудник может создавать только задачи, назначенные на себя;
- сотрудник может обновлять только свои задачи;
- менеджер работает только с задачами своих проектов;
- администратор и верхняя роль приложения имеют полный доступ.

### 6.3. RLS для таблицы `app.users`
Для таблицы `app.users` реализованы политики:
- сотрудник может читать строки пользователей в пределах разрешенных столбцов;
- сотрудник может изменять только собственную запись;
- администратор и верхняя роль приложения имеют полный доступ.

### 6.4. Сопоставление login-ролей и бизнес-пользователей
Для корректной работы RLS создана вспомогательная функция, которая связывает login-пользователей `demo_*` с прикладными пользователями из таблицы `app.users`.

Используемое сопоставление:
- `demo_employee` -> `trinity`
- `demo_manager` -> `tony`
- `demo_admin` -> `hermione`
- `demo_superuser` -> `sherlock`

## 7. Процедуры управления доступом

### 7.1. Порядок предоставления доступа
1. Создается login-пользователь PostgreSQL.
2. Пользователю назначается одна из прикладных ролей `app_*`.
3. Пользователь получает права через механизм наследования ролей.
4. При необходимости доступ дополнительно ограничивается политиками RLS.

### 7.2. Порядок отзыва доступа
1. У пользователя отзывается роль командой `REVOKE`.
2. При необходимости login-пользователь блокируется или удаляется.
3. Выполняется проверка остаточных привилегий.

### 7.3. Регламент аудита
Для контроля действий пользователей в системе используется таблица `app.access_logs`. Также для контроля изменений задач используется таблица `app.task_history`. Периодически должны проверяться:
- состав ролей;
- привилегии ролей;
- политики RLS;
- результаты тестирования доступа.
